## Non-obvious learnings (not in code/commits)
- **Lazy method compilation is the whole basis of the 6d-2a plan:** `Series(T)` only
  compiles methods *actually instantiated* for a given `T`. So new `BoxedSeries`
  variants only need to satisfy the generic arms called on them (deinit/clone/print/
  len/filter); numeric aggregations that don't list them never compile for them.
- **`moveInto`/`expandWithNulls` in `column_reader.zig` switch on `physical_type`** —
  forgetting to add a physical type there makes a column decode to **silently empty**
  (this exact bug cost time on INT96 in 6c). Any new physical handling must touch both.
- **pyarrow writes BOTH `converted_type` (field 6) AND the modern `LogicalType`
  (field 10)** for e.g. unsigned ints — so 6d-1 worked via `converted_type` alone.
  6d-2a still needs to parse field 10 (currently skipped in `SchemaElement.decode`).
- **Design decisions for 6d-2a** (see spec): distinct types + variants; full lossless
  round-trip; rich ops; **Decimal = i256** (≤76 precision); **INT96** decodes to a
  usable `Timestamp(nanos)` and re-emits as INT96 only when `WriteOptions.emit_int96`
  is set (default = modern INT64); deferred nested/VARIANT/GEO → `Raw` fallback.
- **Nested types (LIST/MAP/STRUCT) were split out to a separate spec, 6d-2b** —
  they need repetition-level record assembly + a nested column model we don't have.
- **`Series(T)` currently hard-codes `if (T == String)`**; 6d-2a generalizes this to a
  `@hasDecl` capability convention (`deinit`/`clone`/`format`/`eql`).
